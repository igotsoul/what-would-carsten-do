# "Pair Programming Board" with Carsten Lützen

## Summary
Carsten introduces the Pair Programming Board — a simple visual matrix for tracking how much pair programming a team actually does across a sprint. Sourced from a book by "Jimmy," the technique makes pair programming transparent and measurable, and includes a playful twist that ties it to retrospective warm-ups.

## Methods & Techniques

### Pair Programming Board / Matrix (ab 00:20)
- **What:** A physical board with a matrix of team members on both axes. Each two-hour pairing session between any two developers earns one mark in the corresponding cell.
- **When to use:** When a team says they want to do more pair programming but has no way to track or visualize it. Useful during sprint planning (set a goal) and sprint review/retro (reflect on actuals).
- **Group size:** Development teams of any size
- **Duration:** Ongoing throughout the sprint; reviewed at sprint end
- **How it works:**
  1. Create a grid with team member names on both axes (a symmetric matrix).
  2. Agree on a unit — e.g., every 2 hours of pairing = 1 mark.
  3. Whenever two people pair program for that unit, they add a mark in their shared cell.
  4. At end of sprint, count total marks and compare against any goal set (e.g., "20 hours of pairing this sprint").
- **Source:** https://www.youtube.com/watch?v=mwIY2ceNIqQ&t=20

### Superhero Post-it Tweak (ab 01:45)
- **What:** If the team used the "Draw yourself as a superhero" retrospective starter, use those superhero post-its as the markers in the Pair Programming Matrix instead of plain tally marks.
- **When to use:** When the team has already run a superhero icebreaker in the retro and you want to carry the playful energy into the board.
- **Group size:** Any team that did the superhero starter
- **Duration:** 1–2 minutes setup
- **How it works:** Replace tally marks with the team member's own superhero drawing or post-it. Seeing "Iron Man" and "Retro Man" in the matrix cells adds a fun, personal touch.
- **Source:** https://www.youtube.com/watch?v=mwIY2ceNIqQ&t=105

## Key Principles
- **Transparency as a lever for behavior change** — Making pair programming visible helps teams honestly assess whether they are living up to their stated intentions.
- **Quantify what you want to grow** — Setting a sprint goal for pair programming hours (e.g., 20 hours) gives the team something concrete to aim for and reflect on.
- **Low-tech, high-impact** — A hand-drawn matrix on the wall is enough; the simplicity is a feature, not a limitation.
- **Playfulness increases engagement** — Small creative tweaks (like the superhero post-its) make process artifacts more human and fun to interact with.

## Facilitation Tips
- Introduce the board at sprint planning so the goal is set upfront, not retrofitted at the retro.
- Agree on the unit of measure (e.g., 2 hours = 1 mark) as a team before the sprint starts to avoid ambiguity.
- Use it as a retro input: pull the board into the retrospective to spark a conversation about what enabled or blocked pair programming this sprint.
- The superhero tweak only works if you run the drawing exercise first — don't force-fit it.

## Best Used When
- A team verbally commits to more pair programming but has no visibility into whether it's happening.
- A Scrum team wants to introduce a lightweight agile engineering practice metric without heavyweight tooling.
- You are looking for a low-effort way to create a sprint-level goal around technical collaboration.
