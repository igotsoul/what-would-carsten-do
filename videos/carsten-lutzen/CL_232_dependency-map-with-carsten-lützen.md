# "Dependency Map" with Carsten Lützen

## Summary
Carsten introduces the Dependency Map — a visual tool for mapping dependencies between multiple teams across upcoming sprints. Using a simple grid of teams vs. sprints and a roll of yarn (or virtual equivalent), teams can make cross-team dependencies visible and use that transparency to have informed planning conversations.

## Methods & Techniques

### Dependency Map (ab 00:30)
- **What:** A visual grid with teams on one axis and upcoming sprints on the other. Dependencies between teams are shown using yarn (physical) or lines (virtual), making cross-team planning constraints immediately visible.
- **When to use:** When multiple teams are working in parallel and need to coordinate which features can be built when. Especially useful during PI Planning or multi-team sprint planning sessions.
- **Group size:** Multiple teams (3+ teams shown in example); scales to larger program increments
- **Duration:** Not specified, but suited for planning sessions
- **How it works:**
  1. Set up a grid: teams on one axis, upcoming sprints on the other
  2. Each team places their planned features in the relevant sprint cells
  3. Identify dependencies — where one team's feature relies on output from another team
  4. Use yarn (physical) or lines (virtual) to connect dependent features across the grid
  5. Analyze the resulting map: where are bottlenecks? What needs to move to an earlier or later sprint?
  6. Have transparent conversations with teams (and the business) about re-sequencing features to resolve conflicts
  7. Keep the map updated — it is a living artifact, not a one-time exercise
- **Source:** https://www.youtube.com/watch?v=gBa3AvYspDA&t=30

## Key Principles
- Dependencies are not static — new ones emerge and old ones resolve throughout the increment; the map must be updated continuously
- Physical first, virtual mirror: doing the exercise in person with yarn is highly engaging and tangible; mirroring it on a virtual board keeps it maintainable afterward
- Transparency enables conversation: the map's value is not just coordination, but creating a shared visual that allows honest discussion about trade-offs with teams and stakeholders
- The existence of a complex dependency map is itself an argument for descaling — the messier the map, the stronger the case for reducing inter-team dependencies at a structural level

## Facilitation Tips
- Use a roll of yarn physically to connect dependencies — the tactile, visual nature makes the problem immediately obvious in a way that spreadsheets cannot
- Mirror the physical board virtually immediately after the session so the map can be maintained asynchronously
- Don't treat the output as a fixed plan — schedule recurring check-ins to revisit and update the map as work progresses
- When dependencies force a feature to move sprints, surface this transparently to the business/users rather than silently re-planning; use the map as a conversation starter

## Best Used When
- Multiple Scrum teams are working toward a shared release and need to coordinate sequencing
- A program increment or PI Planning event is being prepared
- Teams are experiencing surprise blockers that could have been anticipated with more upfront visibility
- A coach or Scrum Master wants to make the cost of dependencies visible to trigger a descaling conversation

## Related Concepts
- PI Planning (SAFe) — Carsten acknowledges the tool has associations with SAFe, but advocates for its value independently of that framework
- Descaling / Team Topology thinking — reducing dependencies structurally rather than just managing them
- Virtual whiteboards (Miro, MURAL, etc.) for async dependency tracking
