# Userstory Mapping with Carsten Lützen

## Summary
Carsten introduces User Story Mapping as a visual technique for capturing user stories and priorities while keeping focus on the actual users of a product. The video walks through a practical example — a city-building online game — demonstrating how to brainstorm around user personas, prioritize features, and draw release boundaries to define an MVP and subsequent versions.

## Methods & Techniques

### User Story Mapping (ab 00:30)
- **What:** A collaborative workshop technique for organizing user stories around the people who will actually use the product, creating a visual map that supports both backlog grooming and release planning.
- **When to use:** When starting a new product or feature set, when you need to align a team on priorities and user needs, or when planning incremental releases (MVP, V0.2, etc.).
- **Group size:** Whole team in the room; can be split into sub-groups per user type for larger brainstorms.
- **Duration:** Not specified; implied to be a focused workshop session.
- **How it works:**
  1. Gather everybody in the room and start a brainstorm — any brainstorming technique works, simple or sophisticated.
  2. Identify the key user personas for the product (e.g., Player, Parent, Supporter, Admin).
  3. For each persona, brainstorm what they want to achieve — add all ideas without filtering.
  4. Prioritize stories per persona (e.g., "build a house" before "build a shop" if houses are foundational).
  5. Review and adjust the ordering of personas themselves if needed.
  6. Draw a horizontal line across the map to define the MVP (V0.1) — include only the minimum needed for a testable prototype.
  7. Draw additional lines for V0.2, V0.3, etc. — but Carsten cautions against planning too far ahead since "we cannot see the future."
  8. Use the map to estimate how many sprints each version will require.

## Key Principles
- Keep the MVP as small as possible — the goal is something you can validate and learn from.
- Story mapping is centered on users, not features; always anchor brainstorming to who benefits.
- The visual nature of the map creates shared clarity and alignment across the whole team.
- Version lines should only be drawn for near-term releases — don't try to version-plan everything upfront.
- Prioritization happens at two levels: stories within a persona, and personas relative to each other.

## Facilitation Tips
- Carsten suggests having everyone in the room at once to ensure shared ownership of the map.
- For larger groups, split participants into sub-groups per persona to brainstorm in parallel.
- Keep brainstorming open and non-judgmental first, then prioritize — don't filter during the divergent phase.
- Use a physical line (drawn on the board or wall) to mark release boundaries — it makes the MVP tangible and discussable.

## Best Used When
- Kicking off a new product or major feature initiative where scope is still fuzzy.
- A team needs to align on who the users are and what matters most to each of them.
- You want to move from a flat backlog to a structured, user-centered view.
- Planning is needed for an MVP or first testable prototype without over-committing to future versions.

## Related Concepts
- **MVP (Minimum Viable Product):** The smallest releasable version that enables validated learning.
- **Sprint Planning:** The map feeds directly into estimating how many sprints a given version will take.
- **Personas / User Types:** Identifying and centering user types is a prerequisite step before mapping stories.
- **Product Backlog Refinement:** Story mapping can serve as a structured alternative or complement to standard backlog grooming sessions.
