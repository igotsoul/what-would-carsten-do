# "Problem Story" with Carsten Lützen

## Summary
Carsten introduces the Problem Story format as a way to structure backlog items by explicitly linking a specific problem to a proposed solution. It is the final video in his series on backlog item formats, following Improvement Stories, Job Stories, and User Stories. Viewers will learn the format's strengths, its limitations, and when it works best.

## Methods & Techniques

### Problem Story Format (ab 00:20)
- **What:** A backlog item structure that frames work as "In order to solve [specific problem], we will build [specific solution]." It makes both the problem and the solution explicit in the backlog item itself.
- **When to use:** Best suited for mature products where teams are fixing known issues rather than exploring new user needs. Works well when the problem is already clearly identified.
- **Group size:** Not specified — applies to any product team managing a backlog.
- **Duration:** Not specified.
- **How it works:**
  1. Identify a concrete problem (e.g., "users dropping out at step two of account creation").
  2. Frame the backlog item as: "In order to fix [problem], we will build [solution]."
  3. Brainstorm multiple proposed solutions before committing to one.
  4. Use prototyping or iterative work to validate which solution addresses the problem best.
  5. Document the explored alternatives so the decision rationale is preserved.
  6. Once a solution is chosen, split it into smaller problems and sub-solutions for further refinement.

## Key Principles
- The Problem Story format makes it easy to separate problem identification from solution selection — a valuable discipline in backlog management.
- Documenting explored alternatives alongside the chosen solution creates a useful decision trail for the team.
- Carsten notes that the format lacks user context by design, which can cause teams to lose sight of the *why* — the value they are trying to deliver.
- Not every problem is worth solving; lengthy backlogs often contain problems that don't justify the investment. The format alone doesn't address prioritization.
- The format supports iterative decomposition: once a solution direction is confirmed, it can be broken down into smaller problems and solutions recursively.

## Facilitation Tips
- When coaching teams using Problem Stories, prompt them to add a value statement below the story to compensate for the missing user context — Carsten mentions this is a common addition teams make.
- Encourage teams to list multiple proposed solutions before picking one; this turns the format into a lightweight design thinking exercise.
- Use Problem Stories deliberately for mature products where the problem space is well understood, rather than early-stage discovery work.

## Best Used When
- The team is working on a mature product with known, documented problems.
- The goal is to fix or improve something specific rather than explore new user needs.
- The team wants to preserve a record of solution alternatives considered during refinement.
- Backlog items need to be split into smaller, more actionable pieces.
