# Charts with Carsten Lützen

## Summary
Carsten compares Burn Down and Burn Up charts as tools for tracking sprint and release progress. He argues that Burn Up charts convey richer information — especially around scope changes — and are less psychologically misleading than the Burn Down variant.

## Methods & Techniques

### Burn Down Chart (ab 00:20)
- **What:** A line chart tracking remaining work (%) from sprint start to end, ideally trending toward zero.
- **When to use:** Sprint progress tracking when scope is stable and no mid-sprint additions are expected.
- **Group size:** Team + stakeholders
- **Duration:** Ongoing throughout a sprint
- **How it works:**
  1. X-axis: sprint timeline (start → end); Y-axis: remaining work (100% → 0%)
  2. Expected trend: diagonal line from 100% to 0%
  3. In practice: step function as work completes in chunks
  4. Additions to the sprint cause the line to jump upward
- **Limitation:** An upward spike looks like "minus work was done," which is misleading — it usually means scope was added or new knowledge emerged.
- **Source:** https://www.youtube.com/watch?v=EBdd798Bt_A&t=20

### Burn Up Chart (ab 01:50)
- **What:** A chart showing completed work rising toward a defined scope line, making scope changes explicit rather than hidden in remaining-work distortions.
- **When to use:** Preferred over Burn Down whenever scope changes mid-sprint are likely, or for release-level planning on longer horizons.
- **Group size:** Team + stakeholders
- **Duration:** Ongoing throughout a sprint or release

- **How it works:**
  1. X-axis: timeline; Y-axis: work done (0% → 100%)
  2. A horizontal "scope line" marks the current definition of 100% (e.g., 8 issues or 100 story points)
  3. Completed work is plotted as a rising line
  4. If scope is removed, the scope line drops — the gap between the two lines always reflects actual remaining work
  5. Scope additions raise the scope line visibly, keeping the team informed
- **Also works for:** Release planning over multiple sprints — shows whether the release is growing or shrinking over time.
- **Source:** https://www.youtube.com/watch?v=EBdd798Bt_A&t=110

## Key Principles
- A Burn Down chart conflates scope changes with productivity loss — a rising line looks bad even when it simply reflects new discoveries.
- A Burn Up chart separates "how much did we finish" from "how much is there to finish" — two distinct signals on one chart.
- Scope lines in Burn Up charts should ideally be flat; any movement is an explicit, visible team conversation trigger.
- Charts are communication tools — choose the one that conveys the right story to the team and stakeholders.

## Facilitation Tips
- When introducing Burn Up charts to a team used to Burn Down, frame it as "flipping the chart upside down" — it lowers the cognitive switch cost.
- Use the scope line as a discussion anchor in Sprint Reviews or planning sessions: "Why did this line move?"
- For release planning, display the Burn Up chart over multiple sprints to make release scope drift visible to stakeholders early.

## Best Used When
- A team is experiencing mid-sprint scope changes and the Burn Down chart is causing confusion or demotivation.
- Stakeholders need to understand release progress and scope evolution at a glance.
- Retrospectives or Sprint Reviews where transparency around scope decisions is important.
